# ADR: Authentication System (April 2026)

## Status
Accepted — implemented across §1–§8.

## Context

Trade Navigator previously had no authentication. All API routes were open and all data was globally shared. With the product maturing and Taher needing to demo to others without exposing his real portfolio, a multi-user auth system was required.

Key constraints:
- Mobile-first (Expo React Native) — Supabase mobile SDK introduces bundle complexity
- Demo user must work without a real Supabase account
- DB already live with Taher's data — migration must be non-destructive and idempotent
- No RLS desired — application-layer isolation is sufficient at this scale
- Frontend must never touch Supabase directly (keeps auth logic server-side)

## Decision

### Auth Provider: Supabase (server-side only)
- Supabase handles password hashing, JWT issuance, token refresh
- API server holds the service role key; mobile app never sees it
- All Supabase calls go through `POST /api/auth/*` endpoints

### User identity: UUID strings, no users table in PostgreSQL
- `user_id TEXT NOT NULL` column added to all user-owned tables
- Identity managed entirely by Supabase Auth; PostgreSQL has no FK to a users table
- Avoids cross-DB dependency and keeps schema simple

### Demo user bypass
- Token `"demo-token"` bypasses Supabase and maps to `DEMO_USER_ID = "00000000-0000-0000-0000-000000000999"`
- Hardcoded in `artifacts/api-server/src/lib/constants.ts`
- Seed script (`pnpm seed:demo`) populates a realistic portfolio under this ID

### Token storage: AsyncStorage (mobile)
- `@auth_token` key in AsyncStorage
- Module-level `_token` in `AuthContext.tsx` for synchronous access in api helpers
- No Expo SecureStore dependency (acceptable for MVP; upgrade path clear)

### Taher's account
- UUID `9c3ec9cd-4cbf-4994-a6a6-e147ce4ba7ff` generated during migration
- Stored as `TAHER_USER_ID` in `artifacts/api-server/.env`
- Must be registered in Supabase via POST /api/auth/signup to activate

## Architecture

```
Mobile App
  └── AuthContext  ─── AsyncStorage (@auth_token)
        │
        ├── signIn  → POST /api/auth/signin  → supabase.auth.signInWithPassword
        ├── signUp  → POST /api/auth/signup  → supabase.auth.admin.createUser
        ├── tryDemo → token = "demo-token"   (no server call)
        └── signOut → POST /api/auth/signout → supabase.auth.admin.signOut

  └── PortfolioContext api helpers
        └── Authorization: Bearer <token>  (injected via getAuthToken())

API Server
  └── app.ts: requireAuth middleware gates all /api routes
        ├── "demo-token" → req.userId = DEMO_USER_ID
        └── JWT → supabase.auth.getUser(token) → req.userId = user.id

  └── All route handlers: WHERE user_id = req.userId
```

## Migration

`scripts/src/migrate-add-user-id.ts` ran once to:
1. Add `user_id TEXT NOT NULL DEFAULT '<taher-uuid>'` to all user-owned tables
2. Backfill existing rows
3. Drop the DEFAULT (column now required at insert time)

Tables migrated: `accounts`, `positions`, `activities`, `conversations`, `alerts`,
`order_suggestions`, `portfolio_policy`, `portfolio_snapshots`, `macro_posture`,
`position_flags`, `price_alerts`, `policy_proposals`.

Tables that already had nullable `user_id`: `convictions`, `ips_builder_sessions`
(backfilled with Taher's UUID, made NOT NULL).

## Environment Variables

| Variable | Where | Description |
|----------|-------|-------------|
| `SUPABASE_URL` | api-server `.env` | Supabase project URL |
| `SUPABASE_SERVICE_ROLE_KEY` | api-server `.env` | Server-side service role key (never expose to frontend) |
| `SUPABASE_ANON_KEY` | api-server `.env` | Public anon key (currently unused; reserved) |
| `TAHER_USER_ID` | api-server `.env` | `9c3ec9cd-4cbf-4994-a6a6-e147ce4ba7ff` |
| `EXPO_PUBLIC_DOMAIN` | portfolio-tracker `.env` | API server hostname (optional; defaults to localhost:3001) |

## Trade-offs Rejected

| Option | Reason rejected |
|--------|----------------|
| Supabase mobile SDK in React Native | Bundle size, managed auth state complexity, Supabase client in mobile means anon key in bundle |
| Row-Level Security (RLS) | Requires FK from user_id to auth.users, couples DB to Supabase schema, harder to test locally without Supabase |
| Users table in PostgreSQL | Redundant with Supabase Auth; double source of truth for user identity |
| Separate auth service | Over-engineering for MVP |
| Cookie-based sessions | Mobile-unfriendly |

## Future Work

- Migrate token storage to `expo-secure-store` for production hardening
- Add refresh token handling on 401 responses in api helpers
- Google OAuth flow (server-side redirect already scaffolded at `GET /api/auth/google`)
- Rate limiting on `/api/auth/*` endpoints
